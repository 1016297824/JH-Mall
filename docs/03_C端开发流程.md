### 二、网关路由与 C 端认证

> 设计文档：`docs/design/04_gateway详细设计.md`

> ⚠️ **前置条件**：若依网关 `AuthFilter` 会拦截所有请求并用管理端 JWT 校验，C 端 token 无法通过。必须在此步打通 C 端请求通路，否则后续所有 C 端 API 都返回 401。

#### 2.1 问题背景

现有网关配置（Nacos `ruoyi-gateway-dev.yml`）有以下阻塞点：

| # | 问题 | 影响 | 解决 |
|---|------|------|------|
| 1 | `AuthFilter` 是 GlobalFilter，拦截所有路径 | C 端 token 被当管理端 token 校验 → 401 | 将 `/api/**` 加入白名单跳过 AuthFilter |
| 2 | 无 C 端显式路由 | 请求 `/api/user/profile` 无匹配路由 | 新增 `/api/**` 显式路由 |
| 3 | 无 C 端 JWT 校验 | C 端需登录的接口无保护 | 新增 `MallAuthFilter` |

#### 2.2 方案说明

**不变：** `discovery.locator.enabled: true` 保持开启，管理端仍通过 `/{serviceId}/**` 自动路由访问，`AuthFilter` 校验管理端 JWT。

**新增三样东西：**

1. **C 端显式路由**：`/api/auth/**` → mall-auth、`/api/user/**` → mall-user 等（StripPrefix=0）
2. **AuthFilter 白名单**：加入 `/api/**` 和 `/callback/**`，跳过管理端 JWT 校验
3. **MallAuthFilter**：新建 GlobalFilter（order=-150），仅校验 `/api/**` 路径的 C 端 JWT

#### 2.3 Nacos 配置示例

```yaml
# ruoyi-gateway-dev.yml 新增路由
spring:
  cloud:
    gateway:
      routes:
        # ... 保留若依原有路由（/auth/**, /system/**, /code/**, /schedule/**, /file/**）
        # ===== C 端显式路由 =====
        # StripPrefix=0：C 端 Controller 的 @RequestMapping 包含 /api/ 前缀，不能剥离
        - id: mall-auth
          uri: lb://mall-auth
          predicates:
            - Path=/api/auth/**
        - id: mall-user
          uri: lb://mall-user
          predicates:
            - Path=/api/user/**
        - id: mall-product
          uri: lb://mall-product
          predicates:
            - Path=/api/product/**
        - id: mall-order
          uri: lb://mall-order
          predicates:
            - Path=/api/order/**
        - id: mall-payment
          uri: lb://mall-payment
          predicates:
            - Path=/api/payment/**
        - id: mall-marketing
          uri: lb://mall-marketing
          predicates:
            - Path=/api/marketing/**
        - id: mall-search
          uri: lb://mall-search
          predicates:
            - Path=/api/search/**
        # ===== 回调路由 =====
        - id: mall-payment-callback
          uri: lb://mall-payment
          predicates:
            - Path=/callback/payment/**
          filters:
            - StripPrefix=1

security:
  ignore:
    whites:
      # 若依原生白名单
      - /auth/logout
      - /auth/login
      - /auth/register
      - /*/v2/api-docs
      - /*/v3/api-docs
      - /csrf
      # 新增：C 端路由跳过 AuthFilter，由 MallAuthFilter 校验
      - /api/**
      # 新增：回调路径完全放行
      - /callback/payment/**
```

#### 2.4 过滤器执行顺序

| Order | 过滤器 | 职责 |
|:-----:|--------|------|
| -200 | `AuthFilter`（已有） | 校验所有非白名单路径的管理端 JWT |
| -150 | `MallAuthFilter`（新增） | 校验 `/api/**` 路径的 C 端 JWT |
| -100 | `XssFilter`（已有） | JSON body XSS 过滤 |

`AuthFilter` 看到 `/api/**` 在白名单中直接跳过，`MallAuthFilter` 接手检查。`MallAuthFilter` 中白名单内的匿名接口跳过，其余要求 C 端 JWT。

#### 2.5 启动顺序

```bash
# 更新 ruoyi-gateway-dev.yml 后重启网关
# 然后启动 mall-auth(9301) → 后续商城服务
```

### 三、mall-auth 认证模块

> 设计文档：`docs/design/08_mall-auth详细设计.md`

mall-auth 是 C 端认证微服务（端口 9301），提供用户注册、登录、Token 管理。**必须先于其他商城模块启动，网关 MallAuthFilter 依赖其签发 token。**

#### 3.1 手写业务逻辑（无代码生成）

参考：`docs/design/08_mall-auth详细设计.md` 第 3 章。

| 逻辑 | 说明 |
|------|------|
| TokenService | accessToken（30min 有效期）+ refreshToken（7 天）+ Redis 黑名单 |
| 密码登录 | BCrypt 校验 + 错误计数（5 次错误锁定 30min） |
| 短信登录 | 验证码一次性消费，未注册手机号自动注册 |
| 微信登录 | 微信临时 code 换 sessionKey，首次授权自动创用户 |
| 验证码防刷 | 同手机号 60s 冷却 + 同 IP 每日 10 次 + 同手机号每日 5 次 |
| 密码重置 | 修改后踢所有端下线 `revokeAll(userId)` |
| 账户注销 | 设置 `user_status=2`，踢所有端下线 |
| Token 刷新 | refreshToken 一次性轮换，旧 accessToken 保留 1h 宽限期 |

#### 3.2 C 端 API 清单

| # | 方法 | 路径 | 需登录 | 说明 |
|:-:|------|------|:-----:|------|
| 1 | `register(req)` | `POST /api/auth/users` | 否 | 手机号注册（自动签发 token） |
| 2 | `sendSmsCode(req)` | `POST /api/auth/sms_codes` | 否 | 发送验证码（60s 冷却） |
| 3 | `loginByPassword(req)` | `POST /api/auth/sessions` | 否 | 密码登录 |
| 4 | `loginBySms(req)` | `POST /api/auth/sessions/sms` | 否 | 短信登录 |

| 7 | `loginByWechat(req)` | `POST /api/auth/wechat/sessions` | 否 | 微信登录 |

> 设计文档：`docs/design/06_mall-order详细设计.md` + `03_系统详细设计.md` 第 1.3 节 / 第 4~6 章

#### 6.1 手写业务逻辑

参考：`docs/design/06_mall-order详细设计.md` 第 5~7 章。

| 逻辑 | 说明 |
|------|------|
| 下单编排 | Try-Confirm 变体：①幂等校验 → ②参数校验 → ③ Feign 锁库存 → ④ Feign 锁券 → ⑤本地事务创建订单 + Outbox → ⑥返回 |
| 订单状态机 | 9 个状态 + C 端触发 4 个事件（USER_CANCEL / CONFIRM_RECEIPT / APPLY_REFUND / DELETE） |

| 超时关闭 | 定时任务扫描 WAIT_PAY 且超 30min 的订单，自动取消并发 Outbox `mall:order:cancelled` |
| 自动确认收货 | 发货后 15 天自动 COMPLETED |
| 购物车价格同步 | 列表时 Feign 取 SKU 实时价格和库存，库存不足灰显 |

| 支付单状态机 | UNPAID → PAID / FAILED / CLOSED |
| 退款单状态机 | PROCESSING → SUCCESS / FAILED |
| 渠道适配器 | WechatPayAdapter / AlipayAdapter，统一 `createPay(PayRequest) → PayResponse` 接口 |
| 回调验签 | 网关 IP 白名单 + 签名验证，防重放（nonce 唯一约束 + Redis 去重） |
| 幂等支付 | `userId_orderNo_channelCode` 唯一约束，防止同订单同渠道重复发起 |
| Outbox | 支付成功写 `mall:payment:paid`，退款成功写 `mall:refund:succeeded` |

#### 7.2 C 端 API 清单

| # | 方法 | 路径 | 需登录 | 说明 |
|:-:|------|------|:-----:|------|

| 2 | `getPaymentStatus(paymentId)` | `GET /api/payment/payments/{paymentId}` | 是 | 轮询支付状态 |
| 3 | `createRefund(req)` | `POST /api/payment/refunds` | 是 | 发起退款（Feign 内部调用为主） |

### 十、接口联调验证

#### 10.1 验证策略

各模块完成后，使用 Postman / curl 逐接口验证，按以下维度：

| 维度 | 检查项 |
|------|--------|
| 正常流程 | 典型参数调用，验证响应结构符合设计文档 |
| 异常流程 | 必填参数为空、格式错误、资源不存在、状态不允许 |
| 边界值 | 分页 page=0、size=0、超上限 |
| 幂等性 | 重复提交同一 Idempotent-Key |
| 安全 | 未登录访问需登录接口返回 401 |

#### 10.2 核心链路验证

```bash
# 1. 注册/登录
curl -X POST http://localhost:8080/api/auth/users \
  -H "Content-Type: application/json" \
  -d '{"phone":"13800138000","password":"Pass1234","smsCode":"123456","isPrivacyAgreed":1}'

# → 得到 accessToken, refreshToken

# 2. 商品列表
curl http://localhost:8080/api/product/spus?page=1&size=10

# 3. 商品详情
curl http://localhost:8080/api/product/spus/1

# 4. 加购（需 Authorization）
curl -X POST http://localhost:8080/api/order/cart/items \
  -H "Authorization: Bearer <accessToken>" \
  -H "Content-Type: application/json" \
  -d '{"skuId":1,"quantity":2}'

# 5. 下单（需 Idempotent-Key）
curl -X POST http://localhost:8080/api/order/orders \
  -H "Authorization: Bearer <accessToken>" \
  -H "Idempotent-Key: $(uuidgen)" \
  -H "Content-Type: application/json" \
  -d '{"addressId":1,"source":"cart","cartItemIds":[1]}'

# → 得到 orderNo

# 6. 发起支付
curl -X POST http://localhost:8080/api/payment/payments \
  -H "Authorization: Bearer <accessToken>" \
  -H "Content-Type: application/json" \
  -d '{"orderId":"<orderNo>","channel":"wechat_mp"}'
```
