#### 1.2.5 `mall_product_sku_stock` — SKU 库存表（与 SKU 1:1）

| 字段 | 类型 | 约束 | 说明 |
| --- | --- | --- | --- |
| id | bigint unsigned | PK | 主键，自增 |
| sku_id | bigint unsigned | UK `uk_sku_id` | SKU ID，与 SKU 一对一关系 |
| total_stock | int unsigned | DEFAULT 0 | 总库存 = 可用 + 锁定 + 已售 + 冻结，库存初始化时设置 |
| available_stock | int unsigned | DEFAULT 0 | 可用库存：当前可下单购买的数量 |
| locked_stock | int unsigned | DEFAULT 0 | 锁定库存：用户已下单但未支付时锁定，支付后转为已售，取消后释放回可用 |
| sold_stock | int unsigned | DEFAULT 0 | 已售库存：已支付完成的累计销量 |
| frozen_stock | int unsigned | DEFAULT 0 | 冻结库存：售后申请中的商品暂冻，售后完成后再扣减或归还 |
| is_deleted | tinyint unsigned | DEFAULT 0 | 逻辑删除标志：1=已删除 0=未删除，所有查询需加 `is_deleted = 0` 条件 |
| create_time | datetime | NOT NULL | 创建时间，记录行首次插入的时间 |
| update_time | datetime | NOT NULL | 修改时间，记录行最后一次被修改的时间 |
| version | int unsigned | DEFAULT 0 | 乐观锁版本号，库存扣减使用乐观锁防止超卖 |

**业务约束：** `available_stock + locked_stock + sold_stock + frozen_stock = total_stock`

**库存流转逻辑：**

| 操作 | 可用 | 锁定 | 已售 | 冻结 |
| --- | --- | --- | --- | --- |
| 初始化库存 | +N | | | |
| 下单锁库存 | -N | +N | | |
| 支付完成 | | -N | +N | |
| 取消订单 | +N | -N | | |
| 售后申请 | | | -N（或+N） | +N |
| 售后完成退款 | | | | -N |
| 售后拒绝 | | | +N（或-N） | -N |

### 1.6 Outbox 表

#### 1.6.1 `mall_outbox` — Outbox 消息表（每个需要发消息的服务各自维护本地 Outbox 表）

| 字段 | 类型 | 约束 | 说明 |
| --- | --- | --- | --- |
| id | bigint | PK | 主键，雪花 ID |
| message_id | varchar(64) | UK `uk_message_id` | 消息全局唯一 ID（雪花 ID），消费方用于幂等去重 |
| topic | varchar(128) | NOT NULL | 消息主题，如 `mall:order:created`，详见概要设计 12.4 |
| event_type | varchar(64) | NOT NULL | 事件类型，如 `OrderCreatedEvent`，用于消费方反序列化 |
| aggregate_type | varchar(64) | NOT NULL | 聚合类型，如 `order`，标识事件归属的业务聚合 |
| aggregate_id | varchar(64) | NOT NULL | 聚合 ID，如订单号，用于按业务维度追溯事件 |
| payload | text | NOT NULL | 消息体 JSON，使用稳定 DTO 序列化，不直接序列化数据库实体 |
| status | varchar(16) | NOT NULL DEFAULT 'NEW' | 投递状态：NEW / PENDING / SENT / FAILED |
| retry_count | int | DEFAULT 0 | 已重试次数，超过 3 次标记 FAILED |
| next_retry_time | datetime | | 下次重试时间，用于定时任务按指数退避扫描 |
| scheduled_time | datetime | | 预约投递时间，NULL=立即投递；延迟投递场景（如超时关单）设置此值，调度器到时间后才投递 |
| create_time | datetime | NOT NULL | 创建时间，即业务事件产生时间 |
| update_time | datetime | NOT NULL | 修改时间 |

**索引：** `uk_message_id`(message_id)，`idx_status_next_retry`(status, next_retry_time) — 投递器按状态和重试时间扫描，`idx_scheduled_time`(scheduled_time, status) — 调度器按预约时间和状态过滤待投递记录，`idx_aggregate`(aggregate_type, aggregate_id)，`idx_create_time`(create_time)

**投递逻辑：**

1. 业务本地事务中插入 Outbox 记录（status=NEW）
2. 各服务内置定时任务按分页 + 乐观锁抢占待投递记录：`UPDATE outbox SET status='PENDING' WHERE (scheduled_time IS NULL OR scheduled_time <= NOW()) AND status='NEW' LIMIT 100`
3. 投递 MQ，成功则标记 SENT，失败则递增 retry_count 并计算 next_retry_time
4. retry_count >= 3 标记 FAILED，由补偿任务或人工介入
5. PENDING 超 60s 的记录由定时任务回滚到 NEW
6. 已成功投递超过 7 天的记录由清理任务删除
