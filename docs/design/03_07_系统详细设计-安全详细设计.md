## 七、安全详细设计

### 7.1 数据加密

#### 7.1.1 需加密的字段

| 字段 | 敏感度 | 加密原因 |
| --- | :----: | --- |
| 手机号 | 高 | PII，可关联到自然人 |
| 邮箱 | 高 | PII |
| 身份证号 | 极高 | 法律要求保护 |
| 真实姓名 | 中 | PII |
| 支付回调原始报文 | 中 | 含第三方交易流水号，对账后可加密归档 |

#### 7.1.2 加密算法

采用 **AES-256-GCM**（Galois/Counter Mode），Java 8+ 原生支持，无需额外依赖。

```java
// 加密
Cipher cipher = Cipher.getInstance("AES/GCM/NoPadding");
GCMParameterSpec gcmSpec = new GCMParameterSpec(128, iv);  // 128-bit 认证标签
cipher.init(Cipher.ENCRYPT_MODE, secretKey, gcmSpec);
byte[] encrypted = cipher.doFinal(plainText.getBytes(StandardCharsets.UTF_8));
// 输出: IV(12字节) + 密文 + 认证标签(16字节)

// 解密
cipher.init(Cipher.DECRYPT_MODE, secretKey, gcmSpec);
byte[] decrypted = cipher.doFinal(encrypted);
```

**约束：**

| 项目 | 要求 |
| --- | --- |
| 密钥长度 | 256 bit |
| IV（初始化向量） | 每次加密随机生成 12 字节，与密文一起存储 |
| IV 重用 | 绝对禁止同一密钥下重复使用同一 IV |
| 认证标签 | GCM 自动附加，解密时自动校验完整性 |

#### 7.1.3 密钥管理

密钥由 **Nacos** 统一管理，`mall-auth` 作为密钥持有方，其他服务通过 Feign 调用解密接口，不直接持有密钥。

```
Nacos 配置：
mall.security.aes-key = xxxxxxxx（base64 编码的 32 字节密钥）
```

| 约束 | 说明 |
| --- | --- |
| 密钥存储 | 仅存 Nacos，不进代码仓库、不硬编码 |
| 密钥分发 | mall-auth 持有，其他服务 Feign 调解密 |
| 密钥轮换 | 不主动定时轮换，仅在密钥泄漏时手工更换 |

#### 7.1.4 加密字段查询

采用 **SHA256 哈希辅助列** 实现等值匹配：

```sql
-- 建表时同时建哈希列
phone_encrypted varchar(255)  NOT NULL  -- AES-GCM 加密后的密文
phone_hash      varchar(64)   NOT NULL  -- SHA256(phone + salt)

-- 查询时
SELECT * FROM mall_user WHERE phone_hash = SHA2(CONCAT('13800138000', ?), 256)
```

**约束：**

- salt 全局固定，存 Nacos，不走代码
- 仅支持等值匹配，不支持 LIKE 模糊查询
- phone_hash 必须加普通索引，非唯一索引（SHA256 碰撞概率极低但不为零）

#### 7.1.5 解密调用链路

为避免密钥分散到每个服务，解密统一由 `mall-auth` 提供 Feign 接口：

```
服务A（需要解密手机号）→ Feign → mall-auth.decrypt(encryptedData)
                                → mall-auth 用 Nacos 密钥解密
                                → 返回明文
```

| 约束 | 说明 |
| --- | --- |
| 批量解密 | 支持一次请求传入最多 50 条，避免多次 Feign 往返 |
| 超时控制 | Feign 超时 2s，失败重试 1 次 |
| 缓存 | mall-auth 解密后可选缓存明文 1 分钟（Redis），减少重复解密 |

### 7.2 脱敏策略

> 依据：阿里规范【强制】— 用户敏感数据禁止直接展示，必须对展示数据进行脱敏。

#### 7.2.1 脱敏规则

| 数据类型 | 日志中 | C 端 API | 管理端 | 脱敏规则 | 示例 |
| --- | :-----: | :------: | :------------: | --- | --- |
| 手机号 | ✅ | ✅ | 可查看 | 前 3 后 4 | `138****8000` |
| 邮箱 | ✅ | ✅ | 可查看 | 用户名首字符 +`***` | `z***@example.com` |
| 身份证号 | ✅ | ❌ 不返 | ✅ | 前 3 后 4 | `320***********0012` |
| 真实姓名 | ✅ | ✅ | 可查看 | 保留首字符 | `张*` |
| 银行卡号 | ❌ 不存 | ❌ 不返 | ✅ | 前 4 后 4 | `6222****1234` |
| 支付回调报文 | — | ❌ 不返 | 可查看（审计） | 不脱敏，但加密存储 | — |

#### 7.2.2 实现方式：Jackson + AOP 双层组合

**第一层：API 返回脱敏 — Jackson 自定义序列化**

```java
@Retention(RetentionPolicy.RUNTIME)
@JacksonAnnotationsInside
@JsonSerialize(using = SensitiveSerializer.class)
public @interface Sensitive {
    SensitiveType value();  // PHONE / EMAIL / ID_CARD / NAME / BANK_CARD
}
```

```java
// C 端 DTO 加注解，序列化自动脱敏
public class UserProfileDTO {
    @Sensitive(PHONE)  private String phone;       // "13812348000" → "138****8000"
    @Sensitive(EMAIL)  private String email;       // "zhang@example.com" → "z***@example.com"
    @Sensitive(NAME)   private String realName;    // "张三" → "张*"
    private String nickname;    // 非敏感，不脱敏
}
```

管理端需查看完整数据时，使用独立的 `AdminUserDetailDTO`，**不加 @Sensitive**。

**第二层：日志脱敏 — AOP 切面**

```java
@Aspect
@Component
public class DesensitizeLogAspect {
    @Around("@annotation(com.mall.common.annotation.DesensitizeLog)")
    public Object around(ProceedingJoinPoint pjp) throws Throwable {
        Object[] args = pjp.getArgs();
        Object[] sanitized = DesensitizeUtil.deepCopyAndMask(args);
        log.info("调用 {}.{} 参数: {}",
            pjp.getSignature().getDeclaringTypeName(),
            pjp.getSignature().getName(),
            JSON.toJSONString(sanitized));
        return pjp.proceed();
    }
}
```

#### 7.2.3 脱敏实现类

```java
public class DesensitizeUtil {
    public static String maskPhone(String phone) {
        if (phone == null || phone.length() != 11) return "****";
        return phone.substring(0, 3) + "****" + phone.substring(7);
    }
    public static String maskEmail(String email) {
        if (email == null || !email.contains("@")) return "****";
        int idx = email.indexOf('@');
        return email.charAt(0) + "***" + email.substring(idx);
    }
    public static String maskIdCard(String idCard) {
        if (idCard == null || idCard.length() < 8) return "****";
        return idCard.substring(0, 3) + "***********" + idCard.substring(idCard.length() - 4);
    }
    public static String maskName(String name) {
        if (name == null || name.isEmpty()) return "**";
        return name.charAt(0) + "*";
    }
    public static String maskBankCard(String cardNo) {
        if (cardNo == null || cardNo.length() < 8) return "****";
        return cardNo.substring(0, 4) + "****" + cardNo.substring(cardNo.length() - 4);
    }
}
```

#### 7.2.4 关键约束

| 约束 | 说明 |
| --- | --- |
| 脱敏不可逆 | 脱敏是单向的，不能从脱敏值还原原始值 |
| 日志先脱敏再打印 | AOP 在方法执行前完成脱敏，不能打到明文 |
| DTO 分层 | C 端用带 @Sensitive 的 DTO，管理端用无注解的 AdminDTO |
| 内部 Feign 调用 | 服务间调用不脱敏，传输完整数据，由调用方决定是否脱敏 |
| 审计日志 | 审计日志记录操作人 + 操作对象 ID，不记 PII 明文 |

### 7.3 内部请求头签名

#### 7.3.1 防护目标

防止绕过网关直接调用后端服务、服务间 Feign 直连无鉴权被利用，以及中间人篡改内部请求头（`X-User-Id`、`X-Admin-Id` 等）。

#### 7.3.2 签名算法

采用 **HMAC-SHA256**，JDK 原生支持，零额外依赖。

**请求头定义：**

| 请求头 | 示例 | 说明 |
| --- | --- | --- |
| `X-Internal-Timestamp` | `1715947200` | Unix 秒级时间戳 |
| `X-Internal-Nonce` | `a1b2c3d4e5f6` | 随机 16 位 hex，防重放 |
| `X-Internal-Signature` | `4f8a2c...` | HMAC-SHA256(body\|timestamp\|nonce, secret) |

**签名生成：**

计算逻辑对网关和 Feign 调用方完全一致，仅签名方不同——网关签名覆盖"外部→服务"链路，各微服务的 `Feign RequestInterceptor` 签名覆盖"服务→服务"链路。签名密钥 `mall.security.internal-secret` 存放于 Nacos `application-dev.yml`（共享配置），全集群只配一份。

```java
String payload = timestamp + "|" + nonce;
if (body != null && !body.isEmpty()) {
    payload += "|" + body;
}
Mac mac = Mac.getInstance("HmacSHA256");
mac.init(new SecretKeySpec(secret.getBytes(StandardCharsets.UTF_8), "HmacSHA256"));
String signature = Hex.encodeHexString(mac.doFinal(payload.getBytes(StandardCharsets.UTF_8)));
```

| 签名方 | 实现位置 | 覆盖链路 |
| --- | --- | --- |
| 网关 | `GatewayFilter`（Order 应在 MallAuthFilter 之后注入可信头再签名） | 外部→网关→服务 |
| 各微服务 | `Feign RequestInterceptor`（如 `mall-auth` 调 `mall-user` 时由 mall-auth 的拦截器加签） | 服务→服务 Feign 直连 |

**验签（服务侧）：**

```java
// 1. 检查时间戳偏差（±5 分钟）
long now = System.currentTimeMillis() / 1000;
if (Math.abs(now - Long.parseLong(timestamp)) > 300) {
    throw new InvalidSignatureException("时间戳偏差过大");
}
// 2. 检查 nonce 是否已使用（Redis SETNX）
if (!redis.setIfAbsent("mall:internal:nonce:" + nonce, "1", Duration.ofMinutes(5))) {
    throw new InvalidSignatureException("nonce 重复使用");
}
// 3. 重新计算签名并对比
String expectedSign = hmacSha256(body, timestamp, nonce, secret);
if (!expectedSign.equals(signature)) {
    throw new InvalidSignatureException("签名验证失败");
}
```

#### 7.3.3 Nonce 防重放

| 项目 | 配置 |
| --- | --- |
| 存储 | Redis `SETNX`，key 前缀 `mall:internal:nonce:` |
| TTL | 5 分钟（与时间戳偏差窗口一致） |
| 查询 | O(1)，不随请求量增长 |

#### 7.3.4 关键约束

| 约束 | 说明 |
| --- | --- |
| 签名密钥 | `mall.security.internal-secret`，存放于 `application-dev.yml`（Nacos 共享配置），**所有模块和网关共用同一份**，不进代码仓库 |
| 验签顺序 | 先校验时间戳 → 再校验 nonce → 最后校验签名（短路优化） |
| 网关职责 | 清理外部 `X-*` 头 → 注入可信内部头 → 生成签名 |
| Feign 调用方职责 | `RequestInterceptor` 注入 `X-Internal-*` 签名头，算法与网关一致 |
| 服务侧职责 | 校验时间戳 → nonce → 签名，任意一步失败即拒 |
| 白名单 | 健康检查 `/actuator/health`、回调 `/callback/payment/**` 可免签 |
| 强制要求 | 生产环境必须开启，不允许绕过 |

#### 7.3.5 共享实现（mall-api）

签名和验签是每个模块的通用逻辑，收敛到 `mall-api` 模块 `com.mall.api.infrastructure.security` 包，避免重复实现。

| 类 | 职责 | 激活方式 |
| --- | --- | --- |
| `InnerFeignRequestInterceptor` | 调用方加签：生成 nonce → HMAC-SHA256 → 注入 `X-Internal-*` 头 | `@Component`，引入 `mall-api` 的模块自动生效 |
| `InnerSignatureFilter` | 接收方验签：校验时间戳/Redis nonce 防重放/签名对比 | `@Component` 自动注册，无需额外配置 |

**依赖：** `mall-api` 需引入 `spring-boot-starter-data-redis`（nonce 防重放的 `StringRedisTemplate`）。

**新增错误码：**

| 编码 | HTTP | msg | userTip |
| --- | :--: | --- | --- |
| `A0311` | 401 | 内部请求签名校验失败 | 系统繁忙，请稍后再试 |
| `A0312` | 401 | 内部请求缺少签名头 | 系统繁忙，请稍后再试 |

### 7.4 网关安全链路

#### 7.4.1 三条链路概述

| 链路 | 路由前缀 | 认证方式 | 核心检查 |
| --- | --- | --- | --- |
| 管理端 | 非白名单路径（含 `/{serviceId}/**`） | 若依 token + RBAC | token 有效性 → 角色权限 → 权限码 |
| C 端 | `/api/**` | mall-auth token | token 有效性 → 用户状态 → 黑名单 |
| 回调 | `/callback/**` | 支付平台验签 | 签名 → IP 白名单 → nonce 防重放 |

#### 7.4.2 限流策略

采用 **Sentinel** 集群流控（项目已集成 Spring Cloud Alibaba Sentinel）。

| 限流维度 | 策略 | 说明 |
| --- | --- | --- |
| 全局 QPS | 网关层全局限流 | 防止整体过载，默认 10000 QPS |
| 接口级 QPS | 核心接口独立限流 | 下单 100/s、支付 50/s、搜索 500/s |
| 用户级 | 同一用户 N 秒内最多 M 次 | 防单人刷接口，如注册 60s 内最多 3 次 |
| IP 级 | 单 IP 限流 | 防脚本攻击 |
| 降级 | 触发限流返回 429 +`A0410` | "请求过于频繁，请稍后再试" |

#### 7.4.3 请求头清洗与注入

网关在转发前按以下顺序处理：

```
① 剥离：删除请求中所有 X-User-*、X-Admin-*、X-Internal-*
② 认证：根据路由前缀选择认证链路
    ├─ 非白名单路径 → 校验若依 token → 解析 adminId + 角色 + 权限码
    ├─ /api/**   → 校验 mall-auth token → 解析 userId + memberId
    └─ /callback/** → 验签 + IP 白名单 + nonce 防重放
③ 注入：写入可信内部请求头
    ├─ X-Request-Id: traceId
    ├─ (管理端) X-Admin-Id: xxx, X-Admin-Roles: [...]
    └─ (C端) X-User-Id: xxx, X-Member-Id: xxx
④ 签名：生成 X-Internal-Timestamp / X-Internal-Nonce / X-Internal-Signature
⑤ 转发：路由到目标服务
```

#### 7.4.4 关键约束

| 约束 | 说明 |
| --- | --- |
| HTTPS | 生产环境强制 HTTPS，HTTP 请求直接 403 |
| X-Request-Id | 网关生成 traceId，全链路透传，用于日志关联 |
| 回退不降级 | 认证失败不降级到匿名的任何操作；token 校验失败直接 401 |
| 限流提示 | 被限流返回 HTTP 429 +`A0410`，userTip 含重试时间建议 |
| 签名强制 | 内部请求头签名在生产环境不允许关闭 |
| CSRF | JWT 存 Header 非 Cookie，天然免疫 CSRF；管理端若依自带 CSRF Token |
| URL 重定向 | 网关禁止响应外部 302 到非白名单域名；后端不直接拼接用户输入做跳转 |
