# 一、SPA 404 排查：Vue Router 4 路由名冲突

## 1. 问题现象

仅 `/system/user`（系统管理 → 用户管理）页面显示 SPA 的"找不到网页"404 页。其他同级路由（`/system/role`、`/system/menu` 等）访问正常。

## 2. 排查过程

### 2.1 确认网络请求

打开浏览器 DevTools Network 面板，刷新页面后确认：

| 请求 | 状态 | 说明 |
|------|:----:|------|
| `/getUserInfo` | 200 | 用户信息正常 |
| `/getRouters` | 200 | 路由数据正常 |
| 菜单 JSON 中包含 `user` 子项 | — | path、component、name 字段均正确 |

排除后端 API 问题。

### 2.2 定位到前端路由

SPA 404 页是 Vue Router 的 `/:pathMatch(.*)*` 捕获路由渲染的 `<views/error/404.vue>`（`router/index.ts:51`）。这说明 Vue Router **没有匹配到任何已注册路由**，不是组件加载失败或渲染错误。

### 2.3 追踪动态路由注册流程

```
permission.ts 路由守卫
  → usePermissionStore().generateRoutes()   // 调用后端 /getRouters
    → filterAsyncRouter(rdata, false, true) // 递归转换路由组件
      → filterChildren()                    // 拍平 ParentView
      → loadView(component)                 // 匹配 views/ 下的 .vue 文件
    → rewriteRoutes 返回给守卫
  → accessRoutes.forEach(router.addRoute)   // 注册到 Vue Router
  → return { ...to, replace: true }         // 重新导航
```

关键代码位置：
- 守卫：`permission.ts:21-72`（Vue 3 版 `src/permission.ts`）
- 路由生成：`store/modules/permission.ts:35-54`
- 组件匹配：`store/modules/permission.ts:116-125`（`loadView()`）
- 子路由拍平：`store/modules/permission.ts:86-97`（`filterChildren()`）

### 2.4 后端 getRouters 响应分析

`SysMenuController.getRouters()` 调用 `SysMenuServiceImpl.buildMenus()` 构建路由树。

关键方法 `getRouteName()`（`SysMenuServiceImpl.java:447`）：

```java
public String getRouteName(String name, String path) {
    String routerName = StringUtils.isNotEmpty(name) ? name : path;
    return StringUtils.capitalize(routerName);
}
```

当 `route_name` 为空时，用 `path` 首字母大写作为 Vue Router 的 `name`。

数据库实际数据：

| menu_id | menu_name | path | route_name | 所属 |
|---------|-----------|------|------------|------|
| 100 | 用户管理 | user | (空) | 系统管理 |
| 2007 | 用户账号 | user | (空) | 商城管理→用户管理 |

两个菜单的 `path` 都是 `"user"`，`route_name` 均为空，后端为它们生成的 `name` 都是 `"User"`。

### 2.5 路由注册顺序

在 `permission.ts` 守卫中，`accessRoutes.forEach(router.addRoute)` 按 API 返回顺序注册：

1. **先注册** `/system` → 子路由包含 `{ name: "User", path: "user" }`
2. **后注册** `/mall` → 子路由包含 `{ name: "User", path: "mall-user/user" }`（经 `filterChildren` 拍平后）

## 3. 根因

**Vue Router 4 的 `addRoute()` 维护全局 `nameMap`，添加命名路由时先检查该名称是否已存在，若存在则执行 `removeRoute(name)` 删除旧路由，再添加新路由。**

流程：
1. `/system` 的子路由 `"User"` 注册 → `nameMap["User"]` 不存在 → 正常添加
2. `/mall` 的子路由 `"User"` 注册 → `nameMap["User"]` 已存在 → **删除 `/system/user` 子路由**
3. `/system/user` 路径在路由表中消失 → 落到 `/:pathMatch(.*)*` → 显示 404

而 `/system/role` 等路由正常，因为商城没有任何子路由与其同名。

> `removeRoute` 会递归删除该路由及其所有子路由，并从 `nameMap` 中清除。这是 Vue Router 4 的内部行为，没有配置选项可关闭。

## 4. 修复方案

利用 `sys_menu` 表已有的 `route_name` 字段，给商城菜单设置唯一值：

```sql
UPDATE `ry-cloud`.sys_menu
SET `route_name` = 'MallUser'
WHERE `menu_id` = 2007;
```

`getRouteName()` 优先取 `route_name`，不为空时不再 fallback 到 `path`，路由名变为 `"MallUser"`，不再与系统的 `"User"` 冲突。

> 注意：`route_name` 只影响 Vue Router 的 `name` 属性，不影响页面路径和功能。修改后**必须重启前端 Vite 开发服务器**（`npm run dev`），因为 `import.meta.glob` 和 Vue Router 的 `nameMap` 在模块初始化时固化，仅靠 HMR 无法清除已冲突的内置路由记录。

## 5. 经验教训

### 5.1 root cause 链条

```
getRouters 返回两条 path="user"
  → getRouteName() 均生成 "User"
    → router.addRoute(/mall) 发现 nameMap["User"] 冲突
      → removeRoute("User") 删除 /system/user
        → SPA 404
```

### 5.2 预防措施

- **代码生成器不会覆盖已有记录的 `route_name`**（只 INSERT 不 UPDATE），首次生成后即可此设置
- 所有商城模块生成后，检查是否与系统已有菜单的 `path` 重复，重复则设置 `route_name`
- 若依这个字段本身就是设计用来解决此问题的

### 5.3 排查同类问题的模式

遇到"只有某个页面 404，其他同级页面正常"的场景，排查路径：

1. 确认 API 正常 → 排除后端
2. 检查 `getRouters` 响应中的 `name` 字段 → 排查路由名冲突
3. 确认 `loadView` 能正确匹配到 `.vue` 文件 → 排除组件缺失
4. 检查 `filterChildren` 拍平后的路径 → 排除路径错乱
